# Concurrent Send for AgentSession

## Context

`AgentSession.send()` currently throws if a turn is already in progress:

```ts
// providers/claude/session.ts:271, providers/codex/session.ts:286
if (this._state !== "idle") throw new Error("A turn is already in progress");
```

But both Claude Code and Codex CLIs natively support receiving user messages mid-turn — the CLI internally queues them and either (a) drains them as `<system-reminder>` attachments on the next tool result batch, or (b) starts a new turn coalescing queued items when the current turn ends. Our guard prevents apps from using this UX.

This spec relaxes the guard, exposes a capability flag, and adds queue-cancellation where the upstream CLI supports it — bringing Agentex to feature parity with Claude Code's own TUI for the "type while it's working" workflow.

## Goals

1. Concurrent `send()` is the default API contract — no opt-in. For Claude and Codex (the flagship providers), the call passes straight through to the CLI's native queue.
2. Apps can cancel a queued (not-yet-processed) message by UUID, for providers whose CLI supports it.
3. Library generates UUIDs; apps don't have to manage them. Returned from `send()` for use with `cancel()`.
4. Public API surface stays small: existing `send / interrupt / close` plus one new `cancel(uuid)` method.
5. Apps can read `provider.capabilities` to gate UX, but don't have to know about it to use the API correctly.

## Non-goals

- No `peek` / `list queued` API. App tracks its own unacked sends (it has the UUIDs from `send()`).
- No retraction of messages that have already started processing (mid-turn drain has already injected them; nothing to remove).
- No in-adapter buffering for providers without native concurrent-send. Such providers throw on concurrent `send()`. App can layer its own queue if needed.
- No changes to `interrupt()` semantics (abort the in-flight turn) or `close()`.

## Design

### Public API

```ts
interface SendHandle {
  /** UUID generated by the library, attached to the user message. Use with cancel(). */
  uuid: string;
  /** Resolves with the next TurnResult after the message was written. May be shared with other concurrent sends. */
  result: Promise<TurnResult>;
}

interface AgentSession {
  readonly sessionId: string | null;
  readonly state: SessionState;

  /**
   * Send a user message. Returns the message's UUID synchronously plus a
   * promise for the resulting TurnResult.
   *
   * Callable at any time (including while a turn is in progress) for providers
   * whose CLI supports it natively — the CLI's own queue handles ordering.
   * For providers that don't support concurrent send, calling while !idle
   * throws. Check `provider.capabilities.concurrentSend` to gate UX if needed.
   *
   * If multiple sends are coalesced or drained into one turn by the CLI, their
   * `result` Promises resolve with the same TurnResult object.
   */
  send(message: string): Promise<SendHandle>;

  /**
   * Cancel a previously-sent message that is still queued in the CLI.
   *
   * Always callable. Returns {cancelled: false} when:
   *   - The provider doesn't support per-message cancellation (capabilities.cancelQueuedMessage === false)
   *   - The message has already been dequeued (race lost to mid-turn drain)
   *   - The UUID is unknown to the CLI
   */
  cancel(uuid: string): Promise<{ cancelled: boolean }>;

  /** Interrupt the currently-running turn. Unchanged. */
  interrupt(): Promise<void>;

  /** Terminate session. Unchanged. */
  close(): Promise<void>;
}
```

### Capabilities (descriptive, not opt-in)

```ts
interface ProviderCapabilities {
  // ... existing fields
  /** True if the underlying CLI accepts user messages mid-turn. App may use this to gate "type while working" UI. */
  concurrentSend: boolean;
  /** True if `cancel(uuid)` can remove queued messages on this provider. */
  cancelQueuedMessage: boolean;
}
```

Capability flags describe *what the provider can do*. They do not gate API access — `send()` and `cancel()` are always callable. They are advisory for apps that want to adapt their UI.

### Behavioral contracts

**Claude adapter:**
- `send(msg)` generates UUID with `randomUUID()`, returns `{uuid, result}` immediately. Writes `{type:"user", uuid, message:{role:"user", content:msg}, parent_tool_use_id:null}` to stdin regardless of `_state`. Removes the `_state !== "idle"` guard.
- `result` resolves when the next `result` event arrives. Multiple in-flight sends MAY resolve with the same TurnResult.
- `cancel(uuid)` writes `control_request {subtype:'cancel_async_message', message_uuid: uuid}` and awaits the matching `control_response`. Returns `{ cancelled }` from the CLI's response.

**Codex adapter:**
- `send(msg)` generates UUID locally (not part of Codex protocol but returned to caller for parity), returns `{uuid, result}`. Writes `turn/start` regardless of `_state`. Removes the guard.
- When `send()` is called during an active turn, Codex's `app-server` merges the second `turn/start` into the active turn as a new `userMessage` item — different mechanism from Claude's mid-turn drain but functionally equivalent (the model addresses both messages in one response). Verified end-to-end via the concurrent smoke test against codex-cli 0.130.0.
- `result` resolves on the next `turn/completed` event after the write.
- `cancel(uuid)` returns `{cancelled: false}`. Codex's official protocol bindings (`codex app-server generate-ts`) confirm there's no per-message cancel in the entire `ClientRequest` union — only `turn/interrupt` (kills whole turn), `turn/steer` (inject into active turn), and `thread/rollback` (drop N turns). `capabilities.cancelQueuedMessage: false`.

**Other providers (cursor, gemini, openclaw, opencode, pi):**
- Until audited (Phase 5), keep the existing guard. `send()` throws when called concurrently. `cancel()` returns `{cancelled: false}`.
- Per-provider audit task lifts the guard where the upstream CLI supports it.

**process adapter:**
- No protocol; can't promise anything about concurrent stdin writes. Keep the guard. `concurrentSend: false`.

### TurnResult contract change

Update the JSDoc on `TurnResult` and `send()` to make this explicit:

```
When concurrentSend is true and multiple send() calls occur during a single
turn, the CLI may merge their messages into one turn's processing — in which
case the Promises returned by each send() resolve with the SAME TurnResult
object. Callers cannot assume 1:1 correspondence between send() calls and
TurnResults.
```

### State machine

Today `_state` transitions: `idle → thinking → tool_executing → ... → idle`. With concurrent send, `_state` no longer gates writes. Keep `_state` for observability (UI can show "thinking") but stop using it as a write barrier.

Add to docs: "session.state reflects the most recent observed lifecycle event, not whether send() is callable. With concurrentSend providers, send() is always callable while state is not 'closed'."

## Risks

1. **Protocol corruption if a non-concurrent CLI gets a mid-turn write.** Mitigated by per-provider capability flag and keeping the guard for unverified providers.
2. **TurnResult ambiguity.** Two concurrent `send()` calls may both resolve with the same merged TurnResult. Documented; app needs to track UUIDs locally.
3. **`cancel(uuid)` race.** Between the moment the app decides to cancel and the moment the CLI processes the control_request, the message may already have been dequeued. `{cancelled: false}` is the honest answer — the app should treat it as "too late."
4. **Codex queueing is informal.** We confirmed empirically via transcript that it works, but the JSON-RPC protocol doesn't formally document it. Risk that a Codex CLI update breaks the behavior. Mitigation: integration test that pins behavior, runs on a known-good Codex version.

## Out of scope (future work)

- **Priority sends.** Claude's queue has `now / next / later`. Could expose as `send(msg, {priority})` later; not in v1.
- **`peek` / queue listing.** No protocol surface for Claude; app-side tracking covers most use cases. Revisit if a real need surfaces.
- **In-adapter buffer for non-concurrent providers.** Would require implementing queue, priority, and dispatch logic for each adapter. Defer until an app actually needs it.

## Open questions

1. **`onTurnResult` callback?** With shared TurnResults, the awaited-result pattern gets awkward for apps that fire-and-forget many sends. Maybe add `session.onResult(cb)` later and let callers ignore `handle.result`. Defer — start with the Promise-based API and let real usage decide.

2. **Interrupt on the codex side.** Worth investigating whether `turn/interrupt` on Codex aborts queued items too, or only the active turn. If it nukes queued items, we could expose a `cancelAll()` for Codex. Defer to T21.

3. **Race between send and immediate cancel.** App calls `send()` then `cancel()` in the same tick. We write the user message to stdin, then immediately write the control_request. The CLI processes them in order — by the time it sees `cancel_async_message`, the user message is already in the queue, so cancel succeeds. But if the CLI is mid-drain when the user message lands, it could be consumed before cancel arrives. Document as best-effort with `{cancelled: false}` as the honest "too late" signal.
